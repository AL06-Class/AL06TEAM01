# PRD 1.0

## 문서 목적

이 문서는 온보딩에서 확인한 문제 정의와 팀 협업 기준을 한곳에 모은 1차 제품 요구사항 문서입니다.

- 입력 문서: `docs/recruitment-platform-product-planning-worksheet.md`
- 협업 기준: `README.md`, `AGENTS.md`, `docs/project-rules.md`, `docs/data-guide.md`, `docs/design-system.md`, `docs/firebase-guide.md`
- Firebase SDK 값은 이 문서에 기록하지 않습니다.

## 1. 제품 개요

### 팀 정보

- 팀 이름: 킵온
- 팀원: 한성준, 강지연, 김범수, 손예슬
- 팀장: 한성준

### 핵심 고객

고향에 계신 부모님과 떨어져 살며 바쁜 일상과 육아를 소화하고 있는 3040 직장인 자녀.

### 문제 정의

3040 직장인 자녀는 홀로 계신 부모님이 요양 등급 환자는 아니지만 노화로 일상에 균열이 생긴 상황에서, 멀리서나마 부모님의 안전과 일상 가사·식사를 정기적으로 챙겨드리고 싶어 합니다. 하지만 부모님이 자식에게 미안해하고 낯선 사람을 경계하며 자존심 상해하는 장애물 때문에, 부모님이 다치거나 외로워지지는 않을지 늘 죄책감과 불안을 안고 살아갑니다.

### 핵심 가치

고객의 시간과 거리 때문에 부모를 돌보지 못한다는 자녀의 불안과 죄책감을 생활도움을 통해 안심으로 바꿉니다.

### 핵심 기능

- 월 단위 구독 플랜 선택 기능
- 방문 도우미 신원 신뢰 인증 카드 뷰
- 제공자의 후기 등록 및 신청자의 후기 열람

### 우리만의 핵심 기능

방문 도우미 신원 신뢰 인증 카드 뷰.

## 2. MVP 사용자 흐름

### 화면 1. 안심 구독 시작

- 가치: 자녀의 미안함과 죄책감 해소
- 행동: 부모님의 상황에 맞는 월 구독 모델을 선택한다
- 기능: 월 단위 구독 플랜 선택 기능
- 주요 구성: 부모님 기본 정보, 건강·특이사항 체크리스트, 요금제 선택, 정기 카드 등록, 웰컴 메시지, 첫 방문일 지정
- 다음 화면: 방문 도우미 신원 신뢰 인증 카드 뷰

### 화면 2. 방문 도우미 신뢰 확인

- 가치: 신원 검증된 이웃을 통한 안전
- 행동: 이번 주 부모님 댁에 방문할 동네 이웃 도우미의 신원을 확인한다
- 기능: 방문 도우미 신원 신뢰 인증 카드 뷰
- 주요 구성: 도우미 프로필 요약 카드, 신원 검증 배지, 과거 평점과 후기, 방문 약속 시간, 예정 업무 요약
- 다음 화면: 타임라인형 부모님 일상 안심 리포트 피드

### 화면 3. 방문 리포트 확인

- 가치: 문제가 해결되었는지 확인
- 행동: 멀리서 부모님이 오늘 어떤 도움을 받았는지 확인한다
- 기능: 제공자의 후기 등록 및 신청자의 후기 열람
- 주요 구성: 사진 포함 수행 타임라인, 어르신 건강·심리 컨디션, 도우미 한 줄 코멘트, 맞춤형 추가 케어 제안
- 다음 화면: 지난 방문 히스토리 보관함 또는 원클릭 추가 결제 화면

## 3. 협업 운영 기준

### 역할 분담

- 팀원은 기획, 요구사항, 기준 결정, 결과 검토를 담당합니다.
- AI는 코드 수정, 문서 수정, 실행 확인, 빌드 확인, 커밋, PR 생성을 담당합니다.
- 팀장은 PR 최종 병합과 배포를 담당합니다.

### 작업 방식

- 앱은 React, TypeScript, Vite 기반 SPA 구조를 유지합니다.
- 역할 기반 화면 분기를 유지합니다.
- 작업 단위는 역할 화면, 공통 구조, 데이터/Firebase, 문서 수정 단위로 구분합니다.
- 유저 플로우는 가능한 독립적으로 구현하고, 중복 구현을 감수해 병합 충돌을 줄입니다.
- 데이터 이름 사전은 공유 기준으로 사용합니다.

### Git / PR

- AI 작업은 개인별 작업 브랜치에서 진행합니다.
- 브랜치 이름은 `teamMember/이름/작업명` 형식을 사용합니다.
- 모든 기능 브랜치는 `main` 병합 전에 PR을 만듭니다.
- PR은 AI가 작성하고, AI 리뷰 결과를 참고해 팀장이 최종 승인합니다.
- 병합은 PR 단위로 하나씩 진행합니다.

### AI가 먼저 확인해야 하는 작업

- 파일 삭제
- 대규모 파일 이동
- 새 라이브러리 추가
- SPA 구조 또는 역할 기반 화면 분기 변경

## 4. 데이터 기준

- 역할 값은 `helper`, `requester`, `guardian`를 사용합니다.
- 필드명은 영어 `camelCase`를 사용합니다.
- 상태값은 자유 텍스트가 아니라 정해진 값만 사용합니다.
- 새 컬렉션, 필드, 상태값, 역할 값은 구현 전에 `docs/data-guide.md`의 공통 이름 사전에 먼저 추가합니다.
- 화면 컴포넌트와 데이터 접근 코드는 분리합니다.

### 확정 방향

- 레퍼런스: Preply의 명확한 탐색, 카드형 프로필, 눈에 잘 들어오는 CTA 흐름
- 분위기: 깔끔·미니멀
- 메인 컬러: Sage Green(`#709775`)
- 보조 색상: Celadon(`#a1cca5`), Muted Teal(`#8fb996`), Hunter Green(`#415d43`), Carbon Black(`#111d13`), 흰색, 옅은 회색

- 최소 기준을 맞추고 세부 구현은 AI가 보완합니다.

- 기본 색상, 버튼, 카드, 표 기준은 크게 벗어나지 않습니다.

## 6. Firebase 기준

- Firebase 설정값은 `.env`에 저장합니다.
- `.env`는 Git에 올리지 않습니다.
- 공유용 파일은 `.env.example`만 사용하며 실제 값은 넣지 않습니다.
- 지금은 Hosting 중심으로 사용합니다.
- Firestore와 Storage는 필요 시 사용합니다.
- 추후 Functions 등 확장을 고려해 기능별 라우팅 가능한 구조를 유지합니다.
- 배포 전 `bun run build`를 확인합니다.

## 7. 논의할 항목

- 도우미 신원 인증 정보의 실제 검증 범위
- 후기와 방문 리포트에 사진 업로드를 포함할지 여부
- 보안 규칙 검토 시점
